⏺ Now let me propose approaches. The main architectural decision is how to
  structure the communication pattern between orchestrator and workers.

  Approach 1: ZMQ ROUTER-DEALER with Worker Pull (Recommended)

  Workers connect to the orchestrator's ROUTER socket and send "READY" messages.
   The orchestrator maintains a ready-worker queue and dispatches tasks to
  available workers. Workers run tasks in an asyncio thread pool and send
  results back, requesting more work.

  Pros: Natural backpressure, simple flow, workers pull work when ready (no
  overloading). Matches the README description closely.
  Cons: Orchestrator is a single point of failure.

  Approach 2: ZMQ PUB-SUB + PUSH-PULL hybrid

  Orchestrator publishes tasks via PUB, workers filter and pull. Results come
  back via a PUSH-PULL channel.

  Pros: Easy to add topic-based routing later.
  Cons: More complex, harder to do adaptive parallelization since orchestrator
  doesn't control dispatch, no natural backpressure.

  Approach 3: ZMQ ROUTER-ROUTER bidirectional

  Both orchestrator and workers use ROUTER sockets for fully bidirectional async
   communication.

  Pros: Maximum flexibility for complex protocols.
  Cons: Overly complex for this use case, harder to debug.

  ---
  My recommendation: Approach 1 (ROUTER-DEALER with Worker Pull). It's the
  simplest design that gives the orchestrator full control over parallelization
  factor and provides natural backpressure. It maps directly to the Paranoid
  Pirate / Majordomo patterns from the ZMQ Guide, which are battle-tested.

⏺ Design Section 2: Communication Protocol

  The orchestrator runs an async pyzmq ROUTER socket. Here's the message flow:

  Worker Registration:
  1. Worker connects to orchestrator's ROUTER endpoint
  2. Worker sends: [READY, worker_id, supported_task_types]
  3. Orchestrator adds worker to its available-workers pool with
  parallelization_factor=1 (initial conservative value)

  Task Submission (Client → Orchestrator):
  1. Client connects to a separate ROUTER endpoint on the orchestrator
  2. Client sends: [SUBMIT, task_type, task_id, json_payload]
  3. Orchestrator enqueues the task, returns: [ACK, task_id]
  4. Client can later send: [STATUS, task_id] or [RESULT, task_id] to poll

  Task Dispatch (Orchestrator → Worker):
  1. Orchestrator checks: does this worker have free task slots (running <
  parallelization_factor)?
  2. If yes, sends: [TASK, task_id, task_type, json_payload]
  3. Worker deserializes JSON into the Pydantic input model, runs the task

  Task Completion (Worker → Orchestrator):
  1. Worker sends: [RESULT, task_id, status, json_result_payload]
  2. Orchestrator stores result, marks task complete
  3. Orchestrator immediately checks if it can dispatch more work to this worker

  Heartbeat:
  - Workers send [HEARTBEAT, worker_id] every N seconds
  - Orchestrator marks workers as dead after M missed heartbeats
  - Dead worker's in-flight tasks are re-queued (respecting retry policy)

⏺ Design Section 3: Adaptive Parallelization Algorithm

  The orchestrator tracks throughput per worker (tasks completed per time
  window) and adjusts each worker's parallelization factor independently.

  Algorithm:
  1. Start with parallelization_factor = worker.cpu_count
  2. Every evaluation window (e.g. 30 seconds), measure throughput:
  completed_tasks / window_duration
  3. If throughput increased since last window → increment factor by 1, continue
   probing
  4. If throughput plateaued or decreased → scale back by 1, enter "steady" mode
  5. In steady mode, re-probe periodically (e.g. every 5 minutes) by bumping
  factor by 1 to see if conditions changed

  Key details:
  - Per-worker tracking, since different machines may have different capacity
  - Simple additive increase, single-step decrease (AISD) — no aggressive
  scaling
  - Minimum factor is 1 (always at least one task slot)
  - The evaluation window should be configurable

⏺ Design Section 4: Task System

  Base class:
  class Task(ABC):
      @abstractmethod
      def input_model(self) -> type[BaseModel]: ...

      @abstractmethod
      def output_model(self) -> type[BaseModel]: ...

      @abstractmethod
      async def run(self, input: BaseModel) -> BaseModel: ...

  Tasks are registered by name via a simple registry (dict mapping task_type
  string → Task class). Worker config file (YAML) lists which task modules to
  import, which auto-registers them.

  Worker config example:

  tasks:
    - "load_task.LoadTask"
    - "my_app.tasks.ProcessDataTask"

  Retry policy is per-task-type, defined on the Task class:
  class LoadTask(Task):
      max_retries: int = 3
      retry_delay_seconds: float = 1.0

  Result storage: Orchestrator keeps an in-memory dict of task_id → TaskResult
  (status, output payload, error info, timestamps). No persistence for v1 —
  results are lost on orchestrator restart.

  Timeout + retry flow:
  1. Worker starts a task, tracks its start time
  2. If task exceeds timeout_seconds, worker cancels it (asyncio cancel for
  async, thread interrupt for sync) and reports TIMEOUT to orchestrator
  3. Orchestrator increments retry count, re-queues if under max_retries
  4. After exhausting retries, orchestrator marks task as FAILED with the error
  history (each attempt's error/timeout info)
  5. All of this is tracked — orchestrator stores per-attempt records so you can
   see: attempt 1 timed out at 30s, attempt 2 failed with exception X, etc.

  Task states: PENDING → RUNNING → COMPLETED | FAILED | RETRYING

⏺ Design Section 5: LoadTask (Rust / pyo3)

  The LoadTask is a Rust extension for stress-testing the system. It simulates
  CPU, I/O, and memory pressure with configurable parameters.

  Input model (Python side):
  class LoadTaskInput(BaseModel):
      duration_seconds: float = 1.0
      concurrency: int = 1          # how many CPU threads to use
      cpu_load: float = 0.7         # fraction of duration spent on CPU work
      io_load: float = 0.2          # fraction spent on simulated I/O 
  (sleep/disk)
      memory_mb: int = 50           # memory to allocate during execution

  Rust implementation:
  - Releases the GIL via py.allow_threads(|| { ... })
  - Uses rayon for parallel CPU work (matrix math / hashing) across concurrency
  threads
  - Simulates I/O with std::thread::sleep + temp file writes
  - Allocates a Vec<u8> of memory_mb size, touches pages to ensure real
  allocation
  - Logs each phase with timestamps

  Output model:
  class LoadTaskOutput(BaseModel):
      phases: list[LoadPhase]       # log of each phase (type, start, duration)
      total_duration_seconds: float
      peak_memory_mb: int

  Built as a separate crate under load_task/, built with maturin develop during
  dev.
